# 记忆管理技能 (skl-memory-rule)

将用户想保存的信息，经过分类、去重、语义扩展后，以标准格式存入 brain。同时提供自动整理功能，将分散的记忆按语义分类聚合。

---

## 核心思路

brain 靠**语义向量**来匹配，存入的文字质量直接决定搜索精度：
- **太简短**（"喜欢苹果"）→ 语义信息少，搜"你吃什么水果"可能找不到
- **有分类前缀**（"[user] 喜欢苹果"）→ 同类信息聚合，搜索更准
- **适度扩展同义词**（"苹果、果类、水果"）→ 不同表达都能命中
- **去重**（已存过的不重复存）→ 避免噪声干扰搜索结果

**整理时**：由 LLM 根据当前上下文生成搜索 query，搜索相关记忆后手动进行分类整理。

---

## 记忆分类体系

存储时为每条记忆打上标签，便于语义聚合。标签完全自定义，按需命名：

- 常用标签：`[user]`、`[project]`、`[idea]`、`[todo]`、`[learn]` 等
- 可多个标签并存：`[user, habit]`、`[project, bug]` 等
- 按需自由创建，不必拘泥

---

## MCP 工具

使用 `mcp__brain__*` 工具：

| 工具 | 用途 | 参数 |
|------|------|------|
| `store` | 存储单条记忆 | `text`, `hit_ids`（引用已有记忆ID列表） |
| `search` | 搜索相关记忆 | `query`（返回 id 和 text） |

### 记忆属性

每条记忆包含：
- `id`：唯一标识
- `text`：记忆内容
- `timestamp`：创建时间
- `hit_count`：被引用次数（越高越重要）
- `last_hit_at`：最后一次被引用的时间
- `decay_score`：遗忘因子（0-1，越高越不容易被遗忘）

decay_score 计算公式：`cosine × log(hit_count+1) × exp(-λ × 天数)`

## 操作一：保存新记忆

当用户说"记住 XX"或"帮我记一下 XX"时：

### 步骤

1. **理解用户意图**
   - 提取核心事实（去掉"帮我记住"、"记一下"等指令词）
   - 判断分类标签

2. **检查是否重复**
   - 调用 `mcp__brain__search`，用核心内容搜索
   - 如果找到高度相似的记忆，说明已有相关记录
   - 提示用户："已有相似记忆：XXX，是否更新？"

4. **获取已有记忆 ID（用于引用）**
   - 搜索相关记忆获取 ID
   - 调用 `mcp__brain__store` 时传入 hit_ids 建立关联

5. **构造优化后的记忆文本**

   格式模板：
   ```
   [标签1, 标签2] {核心事实的清晰陈述}
   扩展: {3~5个紧密相关的同义词或上位词}
   ```

   示例：
   ```
   用户输入: "记住我喜欢走路"

   存储内容:
   [user, habit] 用户喜欢走路
   扩展: 步行、散步、徒步、有氧运动
   ```

   ```
   用户输入: "记住我叫志远"

   存储内容:
   [user] 用户的名字叫志远
   扩展: 姓名、名字、称呼
   ```

5. **调用 `mcp__brain__store`** 存入
6. **反馈给用户**：
   - 新存：✅ 已记住：[user] 用户喜欢走路
   - 更新：✅ 已更新记忆（原：XXX → 新：XXX）
   - 跳过：ℹ️ 已有相同记忆，无需重复存储

## 操作二：自动整理记忆

当用户说"整理一下我的记忆"或"整理关于XX的记忆"时：

### 步骤

1. **LLM 分析当前上下文**
   - 理解用户当前在做什么项目、讨论什么话题
   - 结合用户历史对话，判断需要整理哪些相关记忆

2. **生成搜索 query 并搜索**
   - 根据分析结果，生成最能召回相关记忆的搜索词
   - 例如：用户正在讨论"这个项目"，query 可能是"项目"、"开发"、"代码"等
   - 调用 `mcp__brain__search` 获取所有相关记忆

3. **LLM 逐条分析并重组**
   - 对每条记忆进行去重、分类、优化表达
   - 每条记忆单独成一条，不合并
   - 优化格式：`[{分类标签}] {清晰的事实陈述} 扩展: {同义词}`

4. **LLM 手动整理**
   - 对每条记忆进行去重、分类、优化表达
   - 每条记忆单独成一条，不合并
   - 优化格式：`[{分类标签}] {清晰的事实陈述} 扩展: {同义词}`
   - 调用 `mcp__brain__store` 存入整理后的新记忆（注意引用关联）

5. **返回结果**：
   ```
   整理完成 ✅
   - 找到相关记忆: N 条
   - 去重后保留: M 条
   - 分类整理:
     - user: X 条
     - project: Y 条
     - ...
   ```

## 同义词扩展指南

扩展时遵循"**紧密相关**"原则，3~5个词，不过度联想：

| 原始信息 | ✅ 合适的扩展 | ❌ 过度扩展 |
|---------|------------|-----------|
| 喜欢走路 | 步行、散步、徒步、有氧运动 | 健康、减肥、大自然 |
| 喜欢苹果 | 苹果、果类、水果、青苹果 | 健康饮食、维生素 |
| 喜欢玩游戏 | 游戏、电子游戏、电玩、手游、PC游戏 | 娱乐、休闲、宅 |
| 叫志远 | 姓名、名字、称呼 | 身份、个人 |

扩展词要能真实提升搜索命中，而不是泛化到毫无区分度的词。

---

## 操作三：增强关联召回（连续召回）

为提升记忆关联度，避免关联变成树状而是网状，每次保存记忆后执行连续召回：

### 步骤

1. **保存新记忆后，再用新记忆内容搜索相关旧记忆**
   - 用新记忆的核心关键词搜索
   - 用同义词再次搜索
   - 找到的旧记忆如果还没有指向新记忆的关联，补充关联

2. **多维度搜索发现关联**
   - 用新记忆文本生成多个不同的搜索query
   - 每次搜索发现潜在关联后，都补充关联

3. **迭代召回**
   - 把召回结果的内容再用于生成新的搜索query
   - 形成迭代扩散，发现更深层的关联

4. **定期回溯更新旧记忆**（可选）
   - 不只新记忆关联旧记忆
   - 也回溯更新旧记忆，给它们加上指向新记忆的关联

---

## 操作四：首次搜索失败时的连续召回规则

当搜索记忆时首次搜索找不到目标，必须**连续召回直到找出为止**：

1. **不放弃原则**：首次搜索失败不意味着记忆不存在，继续尝试
2. **扩展关键词**：用同义词、上位词、相关词重新搜索
3. **多轮迭代**：每轮召回的结果内容可以生成新的搜索query
4. **记录召回路径**：方便后续追踪记忆关联

---

## 操作五：利用decay_score自然淘汰实现网状结构

志远提出的创新方案：利用brain的遗忘机制实现自动化的记忆网络优化：

### 核心思路
- **decay_score 公式**：`cosine × log(hit_count+1) × exp(-λ × 天数)`
- 长时间不被引用的记忆会自然衰减遗忘
- 通过**新记忆重构覆盖旧记忆**，整合相关信息到新记忆

### 操作步骤

1. **保存新记忆时发现相关旧记忆**
   - 用新记忆内容搜索相关旧记忆
   - 收集所有相关旧记忆的信息

2. **用新记忆重构并覆盖**
   - 将旧记忆的关键信息整合到新记忆文本
   - 保存新记忆（hit_ids 引用旧记忆）
   - 旧记忆因为不再被引用，decay_score 会逐渐降低

3. **自然淘汰**
   - decay_score 公式会自动让低关联记忆衰减
   - 记忆网自我优化，越久越精简
   - 不需要双向关联，新记忆自动包含所有相关信息

### 优势
- ✅ 不需要双向关联
- ✅ 利用自然衰减机制淘汰无用记忆
- ✅ 新记忆自动整合相关信息
- ✅ 记忆网自我优化

---

## 经验总结：重构记忆后的召回验证

重构记忆后必须验证能否成功召回：

### 关键经验
1. **首次搜索失败不代表记忆不存在** - 要扩展关键词继续尝试
2. **多维度关键词探索** - 标签词、核心概念、同义词都要尝试
3. **连续召回直到找出为止** - 应用操作四的连续召回规则
4. **记录有效关键词** - 哪些关键词能召回，作为以后召回的参考

### 召回测试步骤
1. 重构保存新记忆后
2. 立即用不同关键词测试召回
3. 如果找不到，扩展关键词继续找
4. 直到找到为止，记住有效关键词
