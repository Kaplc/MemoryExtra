## 1. 需求概述

### 1.1 背景与动机

当前前端采用**原生 JavaScript + 动态 HTML 片段加载**的 SPA 模式，存在以下问题：
- **组件化缺失**：每个页面是独立 HTML+JS 片段，无法复用 UI 逻辑
- **状态管理混乱**：使用全局变量（`currentPage`、`pageCache`、各模块的 Timer），难以追踪状态变化
- **路由手写**：`router.js` 通过 fetch 动态加载 HTML，串行加载 JS，复杂页面切换逻辑难以维护
- **AI 编程效率低**：原生 JS 缺乏类型约束和响应式机制，AI 生成代码容易出错

Vue 3 的**模板三段式结构**（template/script/style）让 AI 更少犯错，生态统一减少选择困难，响应式 API 更符合直觉。

### 1.2 核心功能

| 模块 | 当前实现 | Vue 3 重构目标 |
|------|---------|---------------|
| 路由 | 手写 router.js 动态加载 | Vue Router |
| 状态 | 全局 var 变量 | Pinia store |
| 组件 | HTML 片段 + 全局函数 | SFC 组件 |
| 生命周期 | 手写 cleanup() | onUnmounted |
| HTTP | fetch + 全局 fetchJson | Axios + 封装 |
| 图表 | ECharts 原生调用 | ECharts Vue 封装组件 |
| 控制台 | console.js 全局函数 | ConsoleView 组件 |
| Toast | 全局 toast 函数 | useToast composable |

### 1.3 预期目标

1. **目录结构**：Vue 3 + Vite + Vue Router + Pinia
2. **复用现有资源**：保留 echarts.min.js、common.js 工具函数（提取为 composables）
3. **平稳迁移**：先框架后业务，分模块逐步迁移
4. **构建产物**：与当前目录结构兼容，入口仍是 `web/index.html` 加载构建产物

### 4.6 关键算法

#### 6.1 算法设计

**轮询算法（usePolling composable）**：
- 使用 `setInterval` + `onUnmounted` 清理
- 自动处理组件卸载时的 timer 销毁
- 返回 `start/stop` 方法

**ECharts 响应式算法（useEcharts composable）**：
- 接收 `containerRef`（DOM 引用）
- 在 `onMounted` 初始化，`onUnmounted` 销毁
- 提供 `setOption` 方法给父组件调用

#### 6.2 伪代码/流程

```typescript
// usePolling 实现
function usePolling(callback: () => void, interval: number) {
  let timer: number | null = null;

  const start = () => {
    if (timer !== null) return;
    timer = window.setInterval(callback, interval);
  };

  const stop = () => {
    if (timer !== null) {
      clearInterval(timer);
      timer = null;
    }
  };

  onUnmounted(stop);

  return { start, stop };
}
```

## 6. 思路解析原理

### 6.1 设计模式应用

| 模式名称 | 应用场景 | 为什么使用 | 实现方式 |
|----------|----------|-----------|----------|
| Composition API | 所有业务组件 | 逻辑复用，代码组织清晰 | `<script setup>` |
| Store 模式 | 全局状态 | 替代全局 var 变量 | Pinia |
| Composable 模式 | 公共逻辑 | fetchJson、toast 等工具函数封装 | `useXxx()` 函数 |
| 路由懒加载 | 所有页面组件 | 首屏更快，代码分割 | `() => import('./views/XXX.vue')` |

### 6.2 核心机制原理

#### 7.1 机制概述

Vue 3 的**响应式系统**基于 Proxy 实现，当状态变化时自动追踪依赖并更新 DOM，替代了当前手写 `document.getElementById()` + 手动赋值的方式。

#### 7.2 原理详述

1. **响应式原理**：Vue 3 使用 `Proxy` 代理 `reactive()`/`ref()` 返回的对象，读写操作被拦截，自动追踪依赖
2. **组件卸载**：Vue 自动调用 `onUnmounted` 钩子，无需手写 `cleanup()` 函数
3. **路由懒加载**：Webpack/Vite 代码分割，每个路由组件独立 chunk，首屏只加载必要代码

#### 7.3 关键实现点

| 实现点 | 当前方案 | Vue 方案 |
|--------|---------|---------|
| 状态管理 | `var _modelTimer = null` | `const statusStore = useStatusStore()` |
| DOM 更新 | `element.textContent = 'OK'` | `{{ status }}` 模板绑定 |
| 生命周期 | 手写 `cleanup()` + `onPageLoad()` | `onMounted()` + `onUnmounted()` |
| 路由参数 | 全局 `currentPage` 变量 | `useRoute().params` |
| 轮询清理 | `clearInterval` 在 cleanup | `onUnmounted(stopPolling)` |

### 6.3 技术要点

| 技术要点 | 说明 | 注意事项 |
|---------|------|---------|
| Vite 配置 | 保留 `index.html` 入口，配置输出目录 | 与现有 pywebview 集成兼容 |
| Vue Router 懒加载 | `const Wiki = () => import('./views/WikiView.vue')` | 确保 SSR 不可用时正常 |
| Pinia 持久化 | 可选 `pinia-plugin-persistedstate` | 保持现有行为 |
| ECharts 封装 | 提供 `useEcharts` composable | 确保 resize 监听正确 |
| 类型支持 | 启用 `defineProps`、`defineEmits` 类型推断 | 提升 AI 编程准确性 |

## 7. 优缺点分析

## 10. 总结

本方案将前端从原生 JavaScript SPA 重构为 Vue 3 + Vite + Vue Router + Pinia 技术栈。

**核心收益**：
- AI 编程效率提升（模板结构固定，响应式替代手动 DOM 操作）
- 代码可维护性提升（组件化 + 状态管理）
- 开发体验提升（热更新、类型提示）

**实施策略**：
- 分阶段迁移，先基础设施后业务模块
- 保留现有 UI 样式和 API 接口，确保平稳过渡
- 使用 Composables 封装公共逻辑，减少重复代码

**关键风险**：pywebview 集成兼容性、Vue 响应式与现有 JS 混用
